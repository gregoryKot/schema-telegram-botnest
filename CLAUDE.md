# Правила проекта

## Правила по итогам аудита 2026-07 (см. PROJECT_AUDIT.md)

Каждое правило ниже закрывает конкретный класс найденных проблем. Принцип:
**правило без механизма принуждения не работает** — где возможно, правило
дублируется проверкой в CI/тесте.

### 1. Ничего не мержится с красным CI
Каждый PR проходит: `tsc --noEmit` + `jest` + `eslint` + сборки обоих
фронтендов + vitest + **`migrate deploy` на реальном Postgres** (джоба
`migrations`). Ручной запуск тестов «по памяти» — не защита: до июля
2026 в проекте было 144 зелёных теста, которые никто не запускал перед деплоем.

### 1а. Мержит агент, не владелец
Владелец проекта PR-ы **не мержит**. Агент, открывший PR, сам доводит его до
конца: дожидается CI, при красном — чинит и перезапускает, при полностью
зелёном — снимает драфт и мержит (squash) без дополнительного вопроса.
Мерж в `main` = деплой (Amvera тянет из GitHub автоматически), поэтому
«запушил и ушёл» — не завершённая работа: фикс не существует для
пользователя, пока не смержен. Исключение — PR помечен `do-not-merge`
или в описании явно сказано «не мержить».

### 2. Правило тестов распространяется на фронтенды
«Любой новый код с логикой обязан приезжать с тестом» действует и для
webapp/miniapp (vitest + testing-library), а не только для `src/*.spec.ts`.
Логика = ветвления, вычисления, работа с данными. Чистый JSX/стили — нет.

### 3. Парные файлы правятся парой (webapp ↔ miniapp)
Если файл имеет двойника во втором фронтенде — фикс вносится в **оба в одном
PR**, либо файл переносится в shared-пакет. Найдено 3+ случая, когда фикс
(ты/вы, рефакторинг) доехал только до одного фронтенда. Новый общий код —
сразу в shared, а не копипастой.

### 4. Денормализация и дублированные реестры — только с тестом-сверкой
Любые два места, обязанные совпадать (`USER_DATA_TABLES` ↔ `USER_OWNED_TABLES`
↔ schema.prisma; `mySchemaIds` ↔ `UserSchemaNote`), фиксируются тестом,
который падает при рассинхроне. Read-modify-write по денормализованному
полю — только в `$transaction` или через пересчёт из источника правды.

### 5. Идентичность для rate-limit — только верифицированная
Троттлинг-бакет строится по проверенной подписи (JWT/initData). Всё
неверифицированное бакетируется по IP (или `uid|ip`). Иначе лимит
обходится ротацией фейкового `sub`.

### 6. Новый эндпоинт = DTO с class-validator
Inline-интерфейсы в `@Body()` дают только compile-time иллюзию. Рантайм
валидируется декораторами DTO + глобальным `ValidationPipe({ whitelist: true })`.

### 7. Свободный текст от юзера = кризисный путь
Любая новая фича, принимающая свободный текст (дневник, заметка, письмо),
обязана проходить через кризисную детекцию (`crisisMarkers`) с карточкой
телефона доверия 8-800-2000-122. Детекция — на клиенте, до шифрования.
Терапевтический продукт без кризисного пути — недопустим.

### 8. Продуктовая фича = событие аналитики
Новая user-facing фича приезжает с событием в analytics (иначе её эффект
непроверяем). Гипотезы про онбординг/retention проверяются метриками
D1/D7/D30, а не ощущениями.

**Пайплайн событий** (единственный, не плоди второй): фронт шлёт
`api.trackEvent(name, meta)` (fire-and-forget, оба фронтенда — правило №3),
бэк принимает `POST /api/event` (`TrackEventDto` + `AnalyticsController`
санитизирует meta → `AnalyticsService.track` пишет в `AnalyticsEvent`).
Имена событий — allow-list `src/analytics/analytics.constants.ts`
(`ANALYTICS_EVENTS`); новое событие = имя в этот список + ветка в `sanitizeMeta`
(пропускать ТОЛЬКО известные поля — свободный текст/PII в meta недопустимы,
правило №7) + парная константа/тип на фронте (`shared/src/share/analytics.ts`).

**Каждая новая метрика/событие ОБЯЗАНА появиться в отчёте `/stats`.** Событие,
которого нет в отчёте, — невидимо: его будто нет. Механизм:
- агрегат — метод в `BotAnalyticsService` или в `ProductMetricsService`
  (`count()`/`groupBy` для таблиц, `$queryRaw` для группировки по `meta`);
- вывод — **чистый форматтер** (`formatProductMetrics`/`formatRetentionBlock`
  как образцы), который `/stats` подклеивает к тексту (`getAdminStats` или сама
  команда — при переполнении лимита Telegram 4096 шлётся двумя сообщениями);
  форматтер покрыт тестом, **обязательно и пустое состояние** — на чистой БД
  отчёт не должен показывать «0/NaN/мусор»;
- **язык отчёта — простой, без терминов** («поделились N раз», а не
  «share_card events»; «возвращаются ли новенькие», а не «cohort retention»).
  Пиши так, чтобы понял ребёнок; англицизмы/жаргон — в подписи словами.

Общий фронт-код события (трекинг-хук, share-хелпер) — сразу в `shared/`
(правило №3/№11): образцы — `shared/src/analytics/useCrisisCardTracking`,
`shared/src/share/shareCardWithTracking`.

Механизм принуждения (ревью): «новое событие видно в `/stats`? понятно без
словаря? есть тест на форматтер, включая пустую БД?». Нет — фича не готова.

### 9. eslint-храповик
Ни один PR не увеличивает счётчик ошибок eslint. `no-floating-promises` —
error. Новые `any` не добавляются (`no-explicit-any` как минимум warn);
каст Prisma-делегата на `any` запрещён — чини типы, а не глуши их.

### 10. Лимит размера файла — честный и всеобщий
~150 строк для нового кода, жёсткий потолок 300 (бэкенд и фронтенд).
Компонент реакта >300 строк — выноси хуки/подкомпоненты. Правило «один
сервис = один файл» не означает «один файл на 13 доменов».

Механизм принуждения (правило без гейта не работает — до июля 2026 это
правило было единственным в разделе без проверки, и 76+ файлов пробили
потолок, худший — 3386 строк):
- **`scripts/check-file-size-ratchet.mjs`** (CI, джоба `backend`) —
  пофайловый храповик. Файл из `scripts/file-size-baseline.json` может
  только **уменьшаться**: рост сверх зафиксированного размера роняет CI
  (раздутый файл дробится, а не пухнет дальше). Любой **новый** файл (не в
  бейслайне) не имеет права родиться больше 300 строк.
- Снизил размер — зафиксируй прогресс, чтобы файл нельзя было снова
  раздуть: `node scripts/check-file-size-ratchet.mjs --update`. Долг
  (файлы сверх 300) заморожен в бейслайне и обязан таять, а не расти.
- Исключения: `game/` (заморожён), `dist/`, `*.spec/.test`, `*.d.ts`,
  vendor. Крупный дата-файл (контент) — осознанное исключение через
  `--update`, видно в диффе PR.

### 11. Дубли и мёртвый код — под гейтами (по итогам свипа 2026-07)
Свип нашёл: живые маршруты-дубли (`/api/settings` и client-notes в двух
контроллерах — отвечал тот, чей модуль раньше, у второго терялась
privacy-проверка `therapistShareCards`) и ~20 мёртвых файлов-копий
(старые версии секций, замещённые новыми — HelpSection/ExercisesSection
против PracticeSection). Механизмы:
- **`scripts/check-route-collisions.mjs`** (CI): один маршрут — один
  контроллер, второй регистрант = мёртвый код с расползающейся логикой.
- **`scripts/check-jscpd-ratchet.mjs`** (CI): счётчик дублированных строк
  (≥70 токенов) не растёт. Повторяешь блок — выноси в модуль/хук.
- Заменяя компонент/секцию новой версией — **удаляй старую в том же PR**
  (иначе jscpd считает её «дублем», а следующий редактор правит мёртвую).
- Периодически (раз в квартал / перед крупным рефакторингом):
  `npx knip --include files` по каждому пакету + грep-верификация. knip
  не видит script-теги и лениво-динамические импорты — удалять только
  после грепа.

## Структура

- `src/bot/` — бизнес-логика (потребности, оценки, БД)
- `src/telegram/` — всё связанное с Telegram (команды, кнопки, провайдер бота)
- `src/prisma/` — PrismaService, подключение к БД

## Два фронтенда — один бэк

- `webapp/` — сайт `schemehappens.ru` (логин Google/Telegram, JWT)
- `schema-miniapp/` — Telegram мини-апп `schemehappens.ru/app/` (initData)
- Оба ходят в один NestJS API. Данные **общие** если `userId` совпадает.

Когда совпадает:
- Telegram login на сайте ⇄ мини-апп = один и тот же `userId = telegramId` ✓
- Google login на сайте → отдельный web-only `userId`. Чтобы данные подтянулись из мини-аппа — юзеру надо в `/account` нажать "Привязать Telegram" и потвердить merge.

См. `src/auth/merge.service.ts` — транзакционный перенос userId по всем USER_OWNED_TABLES при подтверждённом merge.

**Новая функциональность** → сначала логика в `bot.service.ts`, потом UI в `telegram.service.ts`.
Не смешивай Telegram-логику с бизнес-логикой.

## Обращение «ты/вы» (ОБЯЗАТЕЛЬНО для любых новых текстов)

У пользователя есть выбор формы обращения — поле `User.addressForm` (`'ty' | 'vy'`,
дефолт при `null` = «ты»). Выбор делается при первом входе и меняется в настройках.
Действует **везде**: бот, уведомления, мини-апп, сайт. Данные общие — форма едина.

**Правило: каждая новая user-facing строка с обращением обязана звучать в обеих формах.**
Пользователь с формой «вы» не должен нигде увидеть «ты» (и наоборот).

Как разводить (что различается: местоимения ты/тебя/тебе/тобой, притяжательные
твой/…, глаголы 2 л. ед.ч., императивы `Отметь`→`Отметьте`):

- **Бот / уведомления** (бэкенд): `src/notification/address-form.ts` → `t(form, 'ты-текст', 'вы-текст')`.
  Тексты уведомлений уже полностью двухформенные, ориентируйся на них.
- **Фронтенды** (React): хук `useTr()` один раз в теле компонента → `const tr = useTr();`,
  затем `tr('ты-вариант', 'вы-вариант')`. Ещё есть `useAddressForm(): 'ty'|'vy'`,
  `useSetAddressForm()`, и чистый `pickForm(form, ty, vy)` для не-React мест.
  Инфраструктура — `schema-miniapp/src/utils/addressForm.tsx` и `webapp/src/utils/addressForm.tsx`.
- **Контент-файлы** (`needData.ts`, `aboutData.ts` и подобные `*Data.ts` с текстом):
  паттерн «билдер от формы» — `export const buildX = (tr) => (…)` где каждая различающаяся
  строка обёрнута в `tr('ты','вы')`, плюс хук-обёртка `useX() = getX(useAddressForm())`.
  Консьюмеры берут данные через `useX()`, а не импортируют константу напрямую.

Что **не** разводить (оставлять как есть):
- нейтральные/безличные/номинативные строки (названия, эмодзи, теги, 1-е лицо «я …»);
- дословные внутренние цитаты (`«…»`, `"…"`) — межличностная речь и самоподдержка,
  сохраняют свой регистр;
- сугубо клинические описания-убеждения, где «вы плохи» звучит нелепо — их лучше
  перефразировать **безлично** (form-agnostic), чем городить вилку.

Проверка перед коммитом: прогнать по обоим `src` grep-свип на остаточные «ты»-формы
(`[Тт]ы`, `[Тт]еб[еяё]`, `[Тт]во[йяеё]`, окончания `ешь/аешь/ишь`), убедиться что
осталось только осознанно-нейтральное/цитаты. Лендинг/статьи/игра сайта — маркетинг,
не привязаны к `addressForm`, их не трогаем.

## Онбординг и очевидность: «откуда это и зачем» (ОБЯЗАТЕЛЬНО)

Любая новая user-facing фича (трекер, дневник, тест, практика, экран) обязана
**до первого действия пользователя** ответить на два вопроса — и ответить
**в контексте**, на самом пути, а не в спрятанном «About»-оверлее, куда надо
ещё догадаться зайти:

1. **Откуда это** — что за сущность и на чём основана (напр. «пять базовых
   эмоциональных потребностей» из схема-терапии, а не безымянный «трекер»).
2. **Зачем это делать** — какой результат/паттерн получит пользователь и через
   какой срок (напр. «через 3–5 дней виден паттерн — что питает, что истощает»).

Правила:

- Объяснение живёт там, где пользователь **и так идёт** (шаг онбординга, первый
  экран фичи), а не только в отдельном разделе. Спрятанный контент = контента
  нет: если до смысла надо докопаться, для новичка его не существует.
- **Одно очевидное главное действие на экран.** Если непонятно «куда идти и что
  жамкать» — это баг дизайна, а не задача пользователя. Не вешай на один элемент
  несколько разных жестов (тап-открыть / драг-оценить / тап-разблокировать) без
  явных, различимых аффордансов и достаточных зон нажатия (цель ≥ 44×44).
- Переиспользуй уже выверенный объясняющий контент (`NEEDS_EXPLAINER`,
  `aboutData` и т.п.), а не пиши третью формулировку — тон и форма ты/вы там
  уже сведены (см. правило про «ты/вы»).

Механизм принуждения (ревью): проверяющий смотрит фичу глазами новичка и
спрашивает — «за 5 секунд понятно, что это, зачем и что нажать первым?». Нет —
фича не готова к мержу.

## Файлы

- Максимум ~150 строк на файл. Если больше — выноси в отдельный сервис.
- Не создавай новые файлы без необходимости. Один сервис = один файл.
- Не дублируй провайдеры: `PrismaModule` глобальный, не регистрируй `PrismaService` в других модулях.

## Одна механика — один компонент (НИКОГДА не реализуй одно и то же дважды)

**Одно и то же действие пользователя = один переиспользуемый компонент/хук.**
Если механика (оценить потребность, выбрать режим/схему, слайдер, список карточек)
встречается на нескольких экранах — она живёт в **одном** месте, а не копируется.
Копипаста взаимодействия — прямой источник рассинхрона: правку вносят в одну копию,
вторая тихо остаётся старой (реальный инцидент: оценка потребности была слайдером
и в `NeedSlider`, и в `NeedTodaySheet`; перевели на тап только первую — экраны
разъехались. Итог: общий контрол `NeedRatingBar`, оба экрана берут его).

**Перед тем как писать новый контрол/слайдер/список/парсер — обязательно `grep`:**
уже есть похожее? Есть — переиспользуй или вынеси общий примитив (компонент/хук/util).
Нет — пиши один раз так, чтобы переиспользовать, а не «пока быстренько сюда скопирую».

Красные флаги на ревью (любой = фича не готова):
- две реализации одного ввода (`onPointerDown`/`calcValue`/`trackRef`/одинаковый JSX
  в разных файлах);
- «поправил в одном месте — в другом осталось по-старому»;
- новый компонент, дублирующий уже существующий по смыслу, вместо переиспользования.

Механизм принуждения: перед добавлением интерактивного элемента — поиск существующего;
при дубле — вынос в общий модуль **в том же PR**. Ревьюер спрашивает: «это уже
где-то реализовано? почему не переиспользовано?». Для дублей между фронтендами —
см. правило №3 (парные файлы / shared-пакет).

## Никаких хардкод-заглушек вместо реальных данных (НИКОГДА)

**Данные, которые пользователь видит как свои (оценки, история, «вчера»,
статистика), берутся ТОЛЬКО из реального источника (API/БД/локальное
хранилище) — никогда из захардкоженной константы.** Заглушка-фолбэк —
прямой обман: на чистом аккаунте показываются чужие/выдуманные цифры, и это
читается как «данные не удалились» (реальный инцидент: константа `YESTERDAY`
`{expression: 4, …}` подставлялась как «вчера», когда реальных данных нет).

Правила:

- Нет реальных данных → показывай **пусто/скелетон/«—»**, а не выдуманное
  число. `real ?? HARDCODED_FALLBACK` для пользовательских данных запрещён.
- Демо/сид-данные не живут в рантайм-коде фронта. Нужен демо-режим — явный
  флаг и явная пометка «пример», не молчаливый фолбэк.
- Что **можно** (это не заглушки данных): `placeholder=""` в полях ввода;
  нейтральный **дефолт-старт** редактируемого контрола (слайдер с середины),
  который пользователь тут же меняет и который не выдаётся за сохранённое.

Механизм принуждения (ревью): «эта цифра/строка — из реальных данных
пользователя? что покажется на чистом аккаунте?». Захардкоженное значение,
которое юзер примет за своё, — фича не готова.

## Загрузка: скелетоны по форме контента, а не спиннеры (ОБЯЗАТЕЛЬНО)

**Пока грузятся данные — показывай плейсхолдер ПО ФОРМЕ будущего контента,
а не крутилку/котика и не пустоту.** Скелетон, повторяющий силуэт экрана
(шапка, карточки, строки списка), убирает сенсорный скачок при появлении
данных (нейроинклюзивность, ADHD/аутизм) и даёт ощущение, что экран уже
«на месте», а не «висит».

Правила:

- Единый примитив — `schema-miniapp/src/components/Skeleton.tsx`: `Skeleton`
  (плашка), `SkeletonCard`, `SkeletonLines`, `SkeletonList`, плюс силуэты
  экранов (`TodayScreenSkeleton`, `ScreenSkeleton`). Не пиши новый шиммер
  инлайном — переиспользуй/дополни примитив (правило «одна механика — один
  компонент»).
- **Форма скелетона совпадает с контентом.** Список практик → `SkeletonList`
  из карточек той же высоты; экран с крупной CTA → большой rounded-блок;
  текст → `SkeletonLines`. Три одинаковых серых прямоугольника на месте
  сложного экрана — это «не та форма», чини.
- Спиннер/«Загрузка…»-текст допустим только для мгновенных действий-кнопок
  (сохранение, отправка), не для загрузки экрана или списка.
- Шиммер уже глушится reduced-motion блоком `index.css` — отдельно
  отключать не нужно.

Механизм принуждения (ревью): «что видно в первую секунду загрузки? силуэт
будущего экрана или крутилка/пустой прямоугольник?». Спиннер вместо
скелетона или скелетон не по форме — фича не готова.

## Тесты (ОБЯЗАТЕЛЬНО для новой логики)

**Любой новый код с логикой обязан приезжать с тестом.** Не «когда-нибудь потом» —
в том же коммите. Тесты — `*.spec.ts` рядом с кодом (`npx jest`), сервис
инстанцируется напрямую с замоканной/поддельной Prisma (примеры:
`bot.analytics.service.spec.ts`, `notes.service.spec.ts`).

**Особый случай — read-after-write.** Если фича *пишет* состояние в одном месте,
а *читает/показывает* его в другом — тестируй **связку сохранение→отображение**,
а не только запись. Тест на «сохранилось ли» проходит зелёным даже когда карточку
потом «не найти». Реальные баги живут именно на стыке: пиши тест «сохранил → нашёл».

Триггеры, при которых read-after-write тест обязателен:
- есть **денормализованное/дублированное состояние**, которое должно совпадать
  (напр. `mySchemaIds`/`myModeIds` ↔ наличие `UserSchemaNote`/`UserModeNote`;
  sequence ↔ реальный `max(id)`) — тестируй, что оно не разъезжается;
- у сохранения есть **побочный инвариант** (заполнил карточку ⇒ она в коллекции);
  инвариант держи в сервисе (нижний слой), а не в UI-кнопке — и тестом фиксируй;
- у одного действия **несколько точек входа** (упражнение открывается из 5 мест) —
  тест бьёт по сервису, чтобы ни один клиент не мог сохранить мимо инварианта.

Каждый пойманный на проде баг → регрессионный тест, воспроизводящий сценарий
(комментарий-ссылка на инцидент сверху файла, как в `notes.service.spec.ts`).

## Безопасность

> Полный security-плейбук — модель угроз, инварианты аутентификации/авторизации,
> анти-паттерны, чеклист перед деплоем, реагирование на инциденты — в [SECURITY.md](SECURITY.md).
> Читать перед изменением auth/api/therapy-кода.

- Все env-переменные читаются через `ConfigService` или `process.env` — не хардкодить значения.
- Данные из callback_data (needId, value) всегда валидируются перед использованием.
- Команды с побочными эффектами (например `/post`) защищены проверкой `ADMIN_ID`.

### Критичные env-переменные
- `BOT_TOKEN` — токен бота. При компрометации: revoke в BotFather → новый токен → инвалидирует ВСЕ initData (юзеры в мини-аппе будут разлогинены, пройдут заново)
- `JWT_SECRET` — подпись access/refresh/merge токенов. При компрометации: ротация → все сессии (web) инвалидируются, merge-токены становятся бесполезны. Юзерам надо залогиниться заново.
- `ENCRYPTION_KEY` — шифрует свободный текст (заметки, дневники, письма, безопасное место). **Никогда не ротируй без re-encryption миграции** — иначе все зашифрованные данные превратятся в мусор. План ротации:
  1. Добавить второй ключ `ENCRYPTION_KEY_NEW`
  2. При чтении пробовать оба ключа (decryptRecord fallback)
  3. Прогнать скрипт: расшифровать всё старым ключом, зашифровать новым, переписать в БД
  4. Удалить старый ключ
- `GOOGLE_CLIENT_SECRET` — клиентский секрет для OAuth. Ротация в Google Cloud Console + обновление env.
- `ADMIN_ID` — Telegram ID единственного администратора. Все админские callback'и проверяют по нему.

### Аудит-события
`SecurityLogService` пишет в логи + DM админу при: merge_confirmed, role_changed, therapist_request_submitted, csrf_blocked, suspicious_initdata (попытка подделать подпись). Если приходит много suspicious_initdata — атака или ротация BOT_TOKEN не докатилась.

### Резервные копии БД (важно для шифрования при компрометации)
Поля в БД зашифрованы AES-256-GCM с ключом из env `ENCRYPTION_KEY`. **Если утечёт бэкап Amvera CNPG + env-snapshot одновременно — данные открыты**.

Что нужно от Amvera (запросить у их саппорта):
1. Шифруются ли резервные копии CNPG at-rest?
2. Где они хранятся (в том же ДЦ, отдельно, у третьей стороны)?
3. Кто имеет к ним доступ (только владелец проекта или сотрудники Amvera)?
4. Encryption keys для бэкапов — ваши или Amvera управляет?

Долгосрочные меры (если хочется паранойи):
- ENCRYPTION_KEY держать **не в env**, а в отдельном secret manager / HSM (для Amvera маловероятно).
- Делать собственные офлайн encrypted dumps (`pg_dump | openssl enc -aes-256-gcm`) и хранить в другом облаке.
- Принять что для российского небольшого хостинга это accepted risk.

### Ротация ENCRYPTION_KEY
1. Сгенерить новый ключ: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`
2. В Amvera env: `ENCRYPTION_KEY_OLD = <старый>`, `ENCRYPTION_KEY = <новый>`. Restart.
3. Запустить `npm run rotate-encryption` (через Amvera exec/SSH).
4. Убрать `ENCRYPTION_KEY_OLD`. Restart.

`decrypt()` поддерживает несколько ключей одновременно — старые читаются как fallback, новые записываются всегда current ключом.

### Удаление аккаунта
`deleteAllUserData` теперь **hard delete** (не soft с deletedAt-флагом):
- DELETE по всем USER_DATA_TABLES + AuthProvider + WebSession + TherapistRequest + сам User
- После транзакции триггер `VACUUM ANALYZE` (не FULL, не блокирует) → быстрее освобождает dead tuples
- Юзер удалён логически мгновенно. Физическое освобождение страниц Postgres'ом — через autovacuum в течение часов.

Для строгого GDPR-уровня right-to-erasure нужен ещё `VACUUM FULL` (лочит таблицы) или disk-level encryption. Не делаем, accepted risk.

## Новый fullscreen-лист или оверлей в webapp (ОБЯЗАТЕЛЬНО)

Любой компонент с `position: fixed, inset: 0` (sheet, экран упражнения, оверлей) **обязан** использовать хук `useHistorySheet` — иначе кнопка «Назад» в браузере уведёт пользователя из приложения.

```tsx
import { useHistorySheet } from '../hooks/useHistorySheet'; // путь от компонента

export function MySheet({ onClose }: { onClose: () => void }) {
  const goBack = useHistorySheet(onClose);
  // ...
  return (
    <div style={{ position: 'fixed', inset: 0, zIndex: 90, ... }}>
      <button onClick={goBack}>Назад</button>   {/* НЕ onClick={onClose} */}
    </div>
  );
}
```

Правила:
- Хук вызывается **один раз** в теле компонента, принимает функцию закрытия (`onClose` / `onBack` / `onDone`).
- Все кнопки «Назад» и «Закрыть» внутри листа используют `goBack()`, **не** `onClose()` напрямую.
- Если лист закрывается после сохранения (`await api.save(); goBack()`), тоже `goBack()`.
- Исключение: внутренние мини-модалки поверх листа (тёмный backdrop для подтверждения) — у них свой локальный `onClose`, хук там не нужен.

**Важно — не использовать `history.pushState` или `window.addEventListener('popstate')` напрямую.**
Хук работает через `useNavigate`/`useLocation` из react-router-dom. Прямой вызов `history.pushState` конфликтует с React Router: при нажатии «Назад» срабатывают оба обработчика одновременно — лист закрывается И роутер уходит на предыдущий раздел.

## Деплой webapp

`webapp/dist/` **не хранится в git** — Amvera сама собирает webapp из исходников через
`RUN npm run build --prefix webapp` в `Dockerfile`.

**Деплой = push в GitHub (`origin/main`). Amvera сама тянет из GitHub и пересобирает** —
отдельный `git push amvera` НЕ нужен. Просто `git push origin main`.

### Мини-апп (`schema-miniapp/`) — важно
Собирается **НЕ** в Docker, а заранее: его `dist/` **коммитится в git**, а Dockerfile
только копирует его в `webapp/dist/app/` (быстрая сборка, без лишнего install/build).
Исходник мини-аппа тоже в git. При изменении исходника мини-аппа:

```bash
npm run build --prefix schema-miniapp   # пересобрать dist
git add schema-miniapp/dist && git commit ...   # закоммитить dist
```

`.dockerignore` исключает исходник мини-аппа из Docker-контекста — поэтому собрать его
в Docker нельзя, только копировать готовую `dist`. Не переключай Dockerfile на сборку
мини-аппа — билд упадёт («Could not resolve entry module index.html»).

Локальная сборка webapp для проверки ошибок TypeScript/Vite:

```bash
cd webapp && node_modules/.bin/vite build
```

## Миграции БД (ОБЯЗАТЕЛЬНО — по итогам инцидента 2026-07-16)

Инцидент: рукописная миграция с сидом (`healthy_adult_phrases`) не заполняла
`updatedAt NOT NULL` → упала на проде при `migrate deploy` → P3009 заблокировал
все миграции → **крашлуп прода на несколько часов**. Ни один существующий чек
(tsc, jest, eslint) SQL не исполняет — битая миграция была невидима до прода.

Правила:

1. **SQL миграции обязан быть исполнен до коммита.** Генерируй через
   `prisma migrate dev` (он сразу применяет к dev-БД). Рукописный/правленый
   SQL — прогони `migrate deploy` на чистой локальной БД. «Выглядит валидно»
   — не проверка: инцидентный SQL был синтаксически корректен.
2. **Ловушка Prisma `@updatedAt`**: колонка в БД — `NOT NULL` **без DEFAULT**
   (значение ставит клиент, не Postgres). Любой сырой `INSERT` в миграции/сиде
   обязан явно заполнять `updatedAt` (`CURRENT_TIMESTAMP`). То же для всех
   NOT NULL-колонок без дефолта.
3. **Сид в миграции — идемпотентный**: `CREATE TABLE IF NOT EXISTS`, вставка
   через `WHERE NOT EXISTS` — чтобы повторный прогон после падения/отката был
   безопасен.
4. **Диагностика падения миграции — с ПЕРВОЙ ошибки.** P3009/P3018 — симптом,
   не причина. Ищи в логах первое падение (по времени из "started at ...") и
   его SQLSTATE — иначе чинишь следствие, а не корень (в инциденте так потеряли
   несколько итераций).
5. Восстановление на Amvera без внешнего доступа к БД: одноразовый
   `prisma db execute --file <recover>.sql` в CMD перед `migrate deploy`
   (см. `prisma/recover-p3009.sql`; в Prisma 7 у `db execute` **нет** флага
   `--schema` — URL берётся из `prisma.config.js`). После стабилизации —
   убрать из CMD.

Механизм принуждения: CI-джоба **`migrations`** — поднимает Postgres 16 и
гоняет `prisma migrate deploy` на пустой БД (дважды). Битый SQL роняет PR,
а не прод.

## Новая таблица с пользовательскими данными — обязательный чеклист

При добавлении любой модели с полем `userId` **обязательно** выполнить все 4 шага:

**1. Удаление при удалении аккаунта И перенос при merge**
Добавить имя модели в **оба** реестра:
- `USER_DATA_TABLES` в начале `src/bot/bot.service.ts` — `deleteAllUserData` очистит её;
- `USER_OWNED_TABLES` в `src/auth/merge.service.ts` (+ правило в `UNIQUE_RULES`,
  если у модели есть unique-констрейнт с userId) — иначе при merge аккаунтов
  данные останутся на старом userId и потеряются для пользователя.
TypeScript проверит имена (`_VerifyTables`), а spec-сверка со schema.prisma
(план: этап 1.1 REMEDIATION_PLAN.md) упадёт, если модель забыта в одном из списков.

**2. Шифрование**
Объявить `EncryptSchema` рядом с методами модели:
```typescript
const MY_SCHEMA: EncryptSchema = { strings: ['text', 'note'], jsonArrays: ['items'] };
```
Использовать `encryptRecord(data, MY_SCHEMA)` при записи и `decryptRecord(row, MY_SCHEMA)` при чтении.
Не шифровать: `id`, `userId`, `createdAt`, `updatedAt`, типы-перечисления (modeId, schemaId, needId).

Механизм принуждения: `src/utils/encryption-coverage.spec.ts` — реестр решений
о шифровании для КАЖДОГО String/Json-поля каждой модели. Новое поле без записи
в реестре роняет jest в CI; `enc`-запись проверяется по исходнику сервиса,
`plain`-запись требует причину. Молча добавить plaintext-колонку со свободным
текстом невозможно (найдено аудитом 2026-07-20: черновики дневников, ответы
YSQ, заявки терапевтов, алиасы клиентов и email в magic-link токенах лежали
в открытом виде).

**3. Каскадное удаление в Prisma**
```prisma
user  User  @relation(fields: [userId], references: [id], onDelete: Cascade)
```

**4. После изменений схемы**
```bash
npx prisma generate
```
