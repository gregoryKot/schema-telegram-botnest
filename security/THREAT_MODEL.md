# Модель угроз SchemaLab

Фаза 0 аудита (см. `AUDIT_PLAN.md`). Описывает, что защищаем, от кого, где
проходят границы доверия и какими путями атакующий может достичь цели.

## 1. Реестр активов

По убыванию чувствительности:

| # | Актив | Где хранится | Чувствительность |
|---|-------|--------------|------------------|
| A1 | Клинический свободный текст: дневники (схема/режим/благодарность), заметки, письма себе, «безопасное место», практики, планы | БД (зашифровано AES-256-GCM), кэш в localStorage | **Критическая** — медицинского уровня |
| A2 | Терапевтические данные: заметки терапевта о клиенте, концептуализация случая | БД (зашифровано) | **Критическая** |
| A3 | Психометрия: ответы и результаты YSQ, рейтинги потребностей, childhood-оценки | БД | Высокая |
| A4 | Идентичности и сессии: JWT, refresh-токены, OAuth-привязки | БД (хеши), httpOnly-cookie, память клиента | Высокая |
| A5 | Граф связей: пары пользователей, связи терапевт↔клиент | БД | Высокая |
| A6 | Админ-контроль: `ADMIN_ID`, одобрение заявок терапевтов, broadcast | env, бот | Высокая |
| A7 | Секреты: `BOT_TOKEN`, `JWT_SECRET`, `ENCRYPTION_KEY`, `GOOGLE_CLIENT_SECRET`, `DATABASE_URL` | env (Amvera), локальный `.env` | **Критическая** |
| A8 | Метаданные: кто, когда, IP, User-Agent сессий | БД, логи | Средняя |

## 2. Модель нарушителя

| # | Профиль | Возможности | Что НЕ может |
|---|---------|-------------|--------------|
| T1 | Аноним из интернета | Любые HTTP-запросы, подделка тела/заголовков/cookie | Валидной подписи Telegram/JWT не имеет |
| T2 | Аутентифицированный рядовой юзер | Всё от T1 + валидная сессия одного аккаунта; атакует данные другого юзера (IDOR, абьюз логики) | Контроль чужого провайдера |
| T3 | Злонамеренный «терапевт» | Всё от T2 + роль THERAPIST (одобрена админом или получена обманом) | — проверяется в аудите |
| T4 | Злонамеренный «клиент» | Цепляется к терапевту по invite-коду, пытается атаковать терапевта или других его клиентов | — |
| T5 | Хост / инфраструктура | Доступ к БД, env, бэкапам Amvera | Расшифровать A1–A2 без `ENCRYPTION_KEY` |
| T6 | Доступ к устройству жертвы | Читает браузерный профиль / localStorage / историю | Серверную сессию (refresh-токен httpOnly, но на устройстве cookie доступна) |

## 3. Границы доверия

| Граница | Что пересекает | Контроль на границе |
|---------|----------------|---------------------|
| B1 Клиент → API | HTTP-запросы | `TelegramAuthGuard` / `JwtAuthGuard`, валидация тела, throttler |
| B2 API → БД | SQL через Prisma | Параметризация; сырой SQL в `merge.service` — белый список идентификаторов |
| B3 API ↔ Telegram | `initData`, бот-команды, исходящие сообщения | HMAC-проверка `initData`/widget; `ADMIN_ID` для админ-команд |
| B4 API ↔ Google/VK | OAuth code/token | `state`-cookie, PKCE (VK), проверка `aud`/`email_verified` |
| B5 API → файловая отдача | `ServeStaticModule` | `exclude: /api/*`; путь не из пользовательского ввода |
| B6 dev → prod | Конфиг, секреты | `NODE_ENV`, отдельные env; `SKIP_AUTH` запрещён в prod |
| B7 Сервер → устройство | access-токен, кэш данных | Токен в памяти; кэш A1 в localStorage чистится при logout |

## 4. Атак-деревья

Корень — цель атакующего; ветви — пути; в скобках — текущая защита (проверяется
в Фазах 3–5).

### AT1 — Прочитать клинические данные другого пользователя (A1/A2)
- IDOR на `/api/*`: подсунуть чужой `id` в path/теле *(скоуп по `userId` из гварда — проверить КАЖДЫЙ эндпоинт)*
- Стать «терапевтом» жертвы без её согласия *(invite-код + `assertRelation`)*
- Подделать `initData`/JWT жертвы *(HMAC / подпись HS256)*
- Смержить аккаунт жертвы в свой *(merge-токен только после доказанного контроля провайдера)*
- SQL-инъекция → дамп таблиц *(Prisma; сырой SQL — белый список)*
- Прочитать localStorage на общем устройстве *(чистится при logout; остаток — пока залогинен)*
- Компрометация БД/бэкапа *(шифрование A1/A2; ключ вне дампа)*

### AT2 — Захватить аккаунт (A4)
- Подделать аутентификацию: `initData` / JWT / Login Widget *(подпись)*
- Украсть refresh-токен *(httpOnly + secure + sameSite=strict + theft-detection по family)*
- Украсть access-токен *(только в памяти, TTL 15 мин)*
- Абьюз merge-флоу *(требуется доказательство провайдера source)*
- CSRF/перехват OAuth-callback, кража `code` *(state-cookie, одноразовость)*
- Предсказать токен/код/nonce *(`crypto.randomBytes` — проверить энтропию всех генераторов)*

### AT3 — Эскалация до THERAPIST или админа (A6)
- Самоназначить роль через API *(`therapistMode` не клиент-записываемый; роль — серверная; `/become-therapist` → 410)*
- Одобрить собственную заявку терапевта *(callback `treq:` сверяет `ADMIN_ID`)*
- Прицепиться к чужим клиентам как фейк-терапевт *(invite-флоу, `assertRelation`)*
- Выполнить админ-команду бота *(`/stats`, `/broadcast` сверяют `ADMIN_ID`)*

### AT4 — Массовая утечка / разрушение (A1, A7)
- SQL-инъекция *(Prisma)*
- Утечка бэкапа + env одновременно *(см. `CLAUDE.md` — вопросы к Amvera)*
- Компрометация `BOT_TOKEN`/`ADMIN_ID` *(ротация, проверка скоупа)*
- DoS до отказа: огромный JSON, исчерпание in-memory мап *(лимит тела 256 КБ, throttler, sweep-таймеры)*

## 5. Цели безопасности (определение «защищено»)

- **Конфиденциальность:** A1–A3 доступны только владельцу и связанному по
  активной relation терапевту. В покое — зашифрованы.
- **Целостность:** данные пользователя пишет только он сам (или терапевт в
  рамках relation). Роли и привязки меняются только по серверной логике.
- **Аутентичность:** каждый запрос привязан к подлинной идентичности
  (подпись Telegram / JWT). Подделка невозможна без секрета.
- **Доступность:** одиночный инстанс устойчив к разумному злоупотреблению
  (rate-limit, лимиты тела, таймауты внешних вызовов).
- **Подотчётность:** чувствительные события (merge, смена роли, reuse токена,
  подделка `initData`) логируются и шлются админу.
