# План внешнего аудита безопасности SchemaLab

Формальная аудит-методология. Цель — не «поискать баги», а **доказуемое
покрытие**: показать, что осмотрена каждая точка входа и каждая граница
доверия, а не «повезло заметить». «Найти все уязвимости» недостижимо как
абсолют — честная цель — систематическое покрытие плюс эшелонированная защита.

Связанные документы:
- `../SECURITY.md` — security-плейбук (инварианты, анти-паттерны, инциденты).
- `THREAT_MODEL.md` — Фаза 0: активы, нарушители, границы доверия, атак-деревья.
- `COVERAGE.md` — Фаза 1: инвентарь точек входа + матрица покрытия.
- `AUDIT_FINDINGS.md` — Фаза 7: реестр находок (создаётся по ходу).

## Принципы

1. **Сначала модель угроз, потом код.** Нельзя оценить защиту, не зная, что защищаем и от кого.
2. **Покрытие, а не интуиция.** Каждый эндпоинт, бот-команда, джоба и граница доверия — в реестре; каждый проходит чеклист.
3. **Состязательное мышление.** Для каждой фичи — «как я это сломаю?», а не «работает ли это».
4. **Статика + динамика.** Чтение кода и живая атака ловят разные классы багов. Нужно и то и другое.
5. **Каждая находка — с PoC.** «Выглядит подозрительно» — не находка. Воспроизводимый эксплойт или конкретная строка — находка.
6. **Assume breach.** Проектируем так, будто периметр уже пробит — минимизируем радиус поражения.
7. **Ретест обязателен.** На каждый закрытый баг — security-регресс-тест.

## Фазы

### Фаза 0 — Скоупинг и модель угроз
Реестр активов по чувствительности; модель нарушителя; границы доверия;
атак-деревья на каждый актив верхнего уровня. → `THREAT_MODEL.md`.

### Фаза 1 — Карта атак-поверхности и матрица покрытия
Полный инвентарь точек входа (HTTP-маршруты, бот-команды и callbacks, cron,
внешние интеграции, раздача статики). Матрица покрытия: строка = вход,
колонки = AuthN / AuthZ / валидация / rate-limit / идемпотентность / вывод /
ошибки / статус. → `COVERAGE.md`.

### Фаза 2 — Автоматическая база (SAST / SCA / секреты)
SCA: `npm audit` + `osv-scanner` по обоим проектам. SAST: `semgrep`
(owasp-top-ten, nodejs, typescript, react), опц. CodeQL. Секреты:
`gitleaks`/`trufflehog` по **всей истории git**. Кастомные grep-паттерны под
анти-паттерны из `SECURITY.md` §8. Триаж ложных срабатываний.

### Фаза 3 — Архитектурный разбор
Аутентификация (3 пути + `TelegramAuthGuard`, жизненный цикл токенов,
merge-флоу). Авторизация (модель скоупинга по `userId` + `assertRelation` —
доказать корректность по дизайну). Криптография (AES-256-GCM, мульти-ключ,
ротация). Сессии, CSRF, CORS. Каждое пересечение границы доверия.

### Фаза 4 — Ручной код-ревью (ядро)
Систематически, эндпоинт за эндпоинтом по матрице из Фазы 1. Auth/crypto-код —
построчно. Taint-анализ: каждый пользовательский вход от точки входа до стока
(БД, `fetch`, ответ, лог, сообщение в Telegram). Сырой SQL в `merge.service` —
отдельно. Бизнес-логика — абьюз-кейсы под проект (merge, therapy, pair,
виртуальные клиенты, удаление аккаунта).

### Фаза 5 — Динамическое тестирование (живая атака)
Требует тест-стенда с БД (сид: 2 юзера, терапевт, пара, виртуальный клиент).
Обход аутентификации, IDOR-свип по всем эндпоинтам, абьюз merge/therapy,
гонки, mass assignment, инъекции, CSRF, DoS, проверка per-user rate-limit.
Фаззинг кривого `initData`/JWT.

### Фаза 6 — Инфраструктура и операционка
HTTP-заголовки (CSP, HSTS, X-Content-Type-Options, frame-ancestors,
Referrer-Policy), TLS, управление секретами и история git, шифрование
бэкапов, гигиена логов, доступ к БД, скоуп бот-токена.

### Фаза 7 — Отчёт, исправление, ретест
Каждая находка: severity (CVSS 3.1), описание, PoC, воздействие, ремедиация.
Трекер исправлений → ретест → security-регресс-тест на каждый баг → обновление
`SECURITY.md` (§14 журнал).

## Непрерывность

- `semgrep` + `npm audit` + `gitleaks` в CI на каждый PR.
- Набор security-регресс-тестов (e2e: IDOR, authz, rate-limit).
- Модель угроз — живой документ: новая фича → обновить модель и матрицу.
- Каденс: лёгкий ре-аудит при крупных изменениях `auth`/`api`/`therapy`.

## Рубрика серьёзности (CVSS 3.1)

| Уровень | CVSS | Что это для SchemaLab |
|---------|------|----------------------|
| Critical | 9.0–10.0 | Полный захват аккаунта, массовая утечка клинических данных, RCE |
| High | 7.0–8.9 | Доступ к данным одного чужого пользователя, эскалация привилегий (THERAPIST/админ) |
| Medium | 4.0–6.9 | Ограниченная утечка, обход части контролей, устойчивый DoS |
| Low | 0.1–3.9 | Хардненинг, мелкие отклонения, неверные коды ответа |
| Info | — | Наблюдения без прямого воздействия |
