# План улучшения системы тестов — v2 (2026-07-20)

Продолжение `TEST_COVERAGE_PLAN.md` (закрыт в PR #66). Исходное состояние после
PR #66: бэкенд 1459 тестов (68% строк / 58.4% веток), webapp 272, miniapp 272,
5 e2e-спеков, 11 инвариант-трипваеров, 8 CI-храповиков.

Аудит 2026-07-20 (покрытие + два независимых обхода: качество фронтенд-тестов,
слабости инфраструктуры) показал: **числа хорошие, но распределение кривое** —
покрытие сконцентрировано в хуках/утилях/сервисах, а самые рискованные стыки
(HTTP-слой терапевтического контура, кризисный UI-путь, платёжные страницы,
shared/) не покрыты вовсе. Принцип прежний: сначала зоны, где баг = катастрофа;
каждое новое правило — с механизмом принуждения.

## Диагноз (факты аудита)

**Бэкенд:**
- 10 из 33 контроллеров имеют юнит-спек. Все `therapy/*` (5 контроллеров,
  privacy-критичные: заметки/концептуализации клиентов) — 0% веток, и **ни один
  therapy/booking/api-маршрут не вызывается в e2e**. Сервисы под ними покрыты,
  но HTTP-обвязка (гварды, DTO-байндинг, параметры) — только статическим
  трипваером присутствия гварда, не поведением.
- `therapy-relations.service` 24% веток, `auth.service` 47% веток,
  `bot.analytics.service` 41% строк, каталоги: api 42.5%, therapy 50.9%,
  site-content 52.9%, donation 59.2%.
- ~13% тестов (171 из 1281) ассертят только `toHaveBeenCalled*` без проверки
  аргументов/результата — тест зелёный, даже если хендлер шлёт мусор.
- 9 спеков рукописно дублируют in-memory fake-Prisma (у трёх — собственный
  `matchesWhere`); семантика where-матчинга может разъезжаться.

**Фронтенды:**
- ~10% файлов имеют тест; компонентных тестов почти нет (webapp 0, miniapp 3
  узких «на один баг»). Покрытие — только хуки/утили.
- **Кризисный путь наполовину слеп**: `detectCrisis()` покрыт отлично, но всё
  после него — нет: CrisisCard, useCrisisCardTracking, и главное — ни один тест
  не проверяет «ввёл кризисный текст в дневник → карточка 8-800 показана»
  (правило №7 CLAUDE.md — это терапевтический продукт).
- Кабинет терапевта в miniapp: `useClientDetail.ts` **488 строк, 0 тестов**
  (webapp-двойник покрыт двумя спеками — прямой дрейф правила №3);
  `therapistClientSheet/` ~3435 строк клинических данных — 0 тестов.
- Платёжный UI (Subscribe/Offer/Donate/BookingPaid/PlanSheet) — 0 тестов.
- YSQ: логика (859 строк хука) покрыта, UI-флоу (~1100 строк) — нет.
- Тестовые двойники дрейфуют: `useYsqTest.test.ts` в webapp и miniapp уже
  разошлись (в каждом не хватает блоков из другого); `check-paired-files` тесты
  не сверяет, jscpd тесты исключает — дрейф никем не ловится.

**Инфраструктура:**
- Фронтенд-coverage не гейтится вообще (vitest без `--coverage` в CI; конфиг
  в webapp мёртвый, в miniapp отсутствует).
- `shared/` (38 файлов, 3431 строка): 0 тестов, нет своего vitest, ни одна
  CI-джоба его не запускает — покрывается случайно, через реэкспорты.
- Имена analytics-событий во фронтах — сырые строки, не сверяются с
  `ANALYTICS_EVENTS` бэкенда: опечатка тихо теряет аналитику.
- Нет трипваера «каждый @Body() — DTO с class-validator» (правило №6 проверяется
  e2e-шно на 2-3 эндпоинтах, не системно).
- Нет mutation-тестирования, flaky-детекции, jest-кэша в CI.

## Этап 1 — Кризисный путь и терапевтический контур (риск: вред максимальный)

**1.1 Кризисный UI-путь end-to-end (оба фронтенда).** Компонентные тесты:
рендер каждого листа свободного ввода (SchemaEntrySheet, ModeEntrySheet,
GratitudeEntrySheet, письма, дневники) → ввод кризисной фразы → `CrisisCard`
с 8-800-2000-122 виден; нейтральный текст → карточки нет; событие
`useCrisisCardTracking` уходит. Прямые тесты `shared/CrisisCard` +
`useCrisisCardTracking`.
*Механизм:* трипваер-спек «каждый компонент, импортирующий `detectCrisis`,
имеет тест с кризисным вводом» (grep-сверка, allowlist сокращается).

**1.2 Therapy HTTP-слой.** e2e-ownership на все 5 therapy-контроллеров:
клиент А не читает данные клиента Б; терапевт после разрыва связи получает
403 (регресс инцидента mode-maps из PR #66); юнит-спеки контроллеров
(паттерн auth-контроллеров: прямая инстанциация + типизированные даблы).
`therapy-relations.service` 24%→75% веток.

**1.3 e2e-охват маршрутов как инвариант.** Расширить ownership-sweep на
оставшиеся user-data ресурсы (diary GET/DELETE, plans, tracker-остатки,
settings, pairs).
*Механизм:* новый трипваер `e2e-route-coverage`: сверяет guarded-маршруты
(категория A из route-auth) со списком путей, встречающихся в `test/*.e2e-spec.ts`;
непокрытый маршрут = упавший тест; allowlist только сокращается.

**1.4 Платёжный контур по HTTP.** e2e: вебхуки Robokassa
(booking/subscription/donation) — подпись, идемпотентный повтор, FAIL при
расхождении суммы (юнит уже есть — поднять на HTTP-уровень), capability-token
отмены.

## Этап 2 — Инфраструктура честности (гейты, без которых прогресс откатится)

**2.1 Фронтенд-coverage-храповик.** Включить `vitest run --coverage` в обе
CI-джобы; `scripts/check-frontend-coverage-ratchet.mjs` по образцу бэкендового
(бейслайн + floors на `src/hooks`, `src/auth`). Оживить конфиг webapp, добавить
в miniapp.

**2.2 `shared/` — полноценный пакет.** Свой vitest + прямые тесты (сейчас 0);
CI-джоба `shared`. Тесты общих хуков переезжают из фронтендов В shared (одна
копия вместо дрейфующих двойников — правило №3 решается устранением, а не
сверкой).

**2.3 Дрейф тестовых двойников.** До переезда в shared: слить разошедшиеся
`useYsqTest.test.ts` (объединение блоков обоих) и внести тестовые twins в
`check-paired-files.mjs`.

**2.4 Общий fake-Prisma для юнитов.** Вынести stateful-фейк
(`test/e2e-support/fake-prisma.ts` как база) в переиспользуемый модуль;
мигрировать 9 рукописных реализаций. Одна семантика where — один источник
правды.

**2.5 Трипваер analytics-синка FE↔BE.** Скрипт/спек: каждый строковый литерал
в `trackEvent('…')` обоих фронтендов ∈ `ANALYTICS_EVENTS`; событие бэкенда,
которое никто не шлёт, — warning. (Паттерн `check-address-form`.)

**2.6 Трипваер DTO (правило №6 системно).** Статический спек: каждый `@Body()`
в контроллерах типизирован классом из `dto/` с class-validator-декораторами;
inline-интерфейс/`any` = красный CI. Allowlist текущих нарушителей — только
сокращается.

## Этап 3 — Качество существующих тестов (чтобы зелёное значило зелёное)

**3.1 Убить «called-only» ассерты.** 171 тест усилить до проверки
аргументов/результата/состояния. Приоритет: payment.controller (9),
auth-flow.service (9), telegram.service (11), auth-контроллеры.
*Механизм:* счётчик weak-assert (grep-скрипт) — храповиком, не растёт.

**3.2 Ветки в чувствительных сервисах.** auth.service 47%→75%,
bot.analytics.service 41%→70%, therapy-client-data 55%→75%. Floors в
`coverage-baseline.json` расширить: + `src/therapy`, `src/api`, `src/booking`
(сейчас floors только auth/utils — остальное может тихо проседать до общего
бейслайна).

**3.3 Компонентные тесты критических UI-флоу.** YSQ (интро→вопросы→результат,
оба фронта), TrackerOverlay (webapp 480 строк — 0 тестов; miniapp 850 строк —
только debounce), miniapp `useClientDetail` (488 строк), платёжные страницы
(subscribe/cancel, honeypot, consent). Поведение, не разметка: пустые состояния,
ошибки API, сохранение.

## Этап 4 — Продвинутое (после 1–3)

**4.1 Mutation testing (Stryker) на золотые зоны:** `src/utils/crypto`,
`crisisMarkers`, `auth.service`, `robokassa`. Сначала разовый отчёт (сколько
мутантов выживает при нынешних тестах), потом порог на эти каталоги. Это
единственный честный ответ на вопрос «тест есть, а ловит ли он?».

**4.2 CI-гигиена.** jest-кэш через actions/cache (сейчас ts-jest компилирует
с нуля каждый ран); вынести fs-only чеки (route-collisions, legacy-name) в
лёгкую джобу без `npm ci`; nightly-прогон с `--detectOpenHandles` и тройным
повтором для flaky-детекции (карантин-лист).

**4.3 Bootstrap-хвосты.** `main.ts`/`app.module`/`filters` — 0%: юнит на
`HttpExceptionFilter` (поведение, не присутствие), smoke на bootstrap-опции
(частично уже статически в hardening-инварианте).

## Чего НЕ делаем (осознанно)

- Не гонимся за 100% строк: телеграм-хендлеры чистой разметки и JSX без логики
  не тестируем (правило №2 CLAUDE.md).
- Никаких снапшот-тестов (хрупкость без смысла; в проекте их нет — так и
  держим).
- Не мокируем то, что можно прогнать честно: фейк-Prisma stateful, а не
  `mockResolvedValue` на каждый вызов.
