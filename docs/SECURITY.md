# Безопасность проекта Schema Happens

Документ для разработчиков (и для будущих сессий Claude). Описывает модель
угроз, инварианты безопасности, чеклисты при изменениях кода и порядок
реагирования на инциденты.

Операционные процедуры (ротация ключей, бэкапы Amvera, удаление аккаунта)
описаны в `CLAUDE.md`, раздел «Безопасность». Здесь — то, что нужно держать
в голове **при написании кода**.

---

## 1. Что мы защищаем — модель угроз

Приложение хранит **клинические данные психического здоровья**:

- дневники (схема / режим / благодарность), заметки, письма себе, «безопасное место»;
- результаты теста YSQ, активные схемы и режимы;
- терапевтические заметки и концептуализацию случая (связь терапевт ↔ клиент).

Это чувствительные персональные данные. Угрозы по убыванию приоритета:

1. **Захват чужого аккаунта** — доступ к чужим дневникам и заметкам.
2. **Утечка данных в покое** — компрометация БД или бэкапа.
3. **IDOR** — авторизованный пользователь читает данные другого по `id`.
4. **Утечка секретов** — `BOT_TOKEN` / `JWT_SECRET` / `ENCRYPTION_KEY`.
5. **DoS** — единственный инстанс, исчерпание ресурсов / CPU / памяти.

**Атакующий по умолчанию:** любой человек с Telegram или браузером. Он может
слать произвольные HTTP-запросы, подделывать тело, заголовки и cookie. Он
**не** контролирует чужой Telegram-аккаунт и чужой Google/VK. Граница доверия
проходит именно здесь: всё, что приходит от клиента, — враждебно, пока не
доказано обратное подписью.

## 2. Инварианты аутентификации

Три пути входа, все сводятся к `req.telegramUserId` (number) либо
`req.webUser.userId` (BigInt):

| Путь | Что присылает клиент | Как проверяется |
|------|----------------------|-----------------|
| Мини-апп | заголовок `x-telegram-init-data` | HMAC initData, свежесть ≤ 1 ч |
| Веб (JWT) | `Authorization: Bearer <jwt>` | подпись HS256, issuer/audience |
| Telegram Login Widget | тело POST | HMAC по `SHA256(bot_token)` |

Унифицирует всё `TelegramAuthGuard` (`src/api/telegram-auth.guard.ts`).

**Инварианты, которые НЕЛЬЗЯ нарушать:**

- Каждый контроллер с пользовательскими данными — под `@UseGuards(TelegramAuthGuard)`
  (либо `JwtAuthGuard` для чисто-веб эндпоинтов). Контроллер без гварда = открытый
  эндпоинт для всего интернета.
- `SKIP_AUTH` действует только при `NODE_ENV !== 'production'`. Проверку
  `process.env.NODE_ENV !== 'production'` в гварде **никогда не убирать** —
  иначе утёкший/ошибочный env даёт полный захват любого пользователя.
- Свежесть initData в гварде и в `verifyTelegramWebAppData` должна совпадать
  (сейчас 3600 с). Рассинхрон = разное окно replay-атаки.
- Сравнение HMAC/подписей — только `crypto.timingSafeEqual`, никогда `===`.
- JWT при verify — всегда явный `algorithms: ['HS256']`. Без этого возможна
  alg-confusion и подделка через `alg: none`.
- **merge-токен** — единственное доказательство намерения объединить аккаунты.
  Выдаётся только после реальной проверки провайдера. Нельзя выдать merge-токен
  для `source = чужой userId`, не доказав контроль над провайдером source.

## 3. Авторизация и защита от IDOR

Аутентификация говорит «кто ты». Авторизация — «можно ли тебе вот это». IDOR —
самый вероятный класс багов в этом проекте.

- Любая выборка данных фильтруется по `userId` **из гварда**, не из тела/параметра.
- `id` из path (`parseId`) — это просто число, а не «доверенный» идентификатор.
  Запрос всё равно обязан быть `where: { id, userId: <из гварда> }`.
- Удаление и обновление — `deleteMany` / `updateMany` со скоупом по владельцу:
  `where: { id, userId }`. **Никогда** `delete({ where: { id } })` без `userId`.
- Терапевт ↔ клиент: каждый аксессор данных клиента **первой строкой** вызывает
  `assertRelation(therapistId, clientId)` (`src/therapy/therapy.service.ts`).
  Роль `THERAPIST` сама по себе не даёт доступа к произвольному клиенту — нужна
  активная `TherapyRelation`.
- Виртуальные клиенты имеют отрицательный `id` (`-relation.id`) — `assertRelation`
  обрабатывает оба случая, не обходить его.

## 4. Валидация входных данных

Всё, что приходит от клиента, валидируется **на границе** (в контроллере) до
передачи в сервис.

- **Числа:** `Number.isInteger(x)` + проверка диапазона.
- **Массивы:** `Array.isArray(x)` + ограничение длины + проверка каждого элемента.
  Без лимита длины массив на 10 000 элементов = DoS.
- **Даты:** regex формата ISO до `new Date(...)`. `new Date(мусор)` даёт
  `Invalid Date` → Prisma падает с 500.
- **Строки:** `.slice(maxLen)` перед записью.
- **callback_data из Telegram** (`need:safety`, `rate:safety:7`): `needId`,
  `value`, `code` валидируются перед использованием.
- Никогда не доверять `body.userId`, `body.role`, `body.therapistMode` и т.п. —
  привилегированные поля выставляются только сервером.

## 5. Шифрование данных

Полная процедура — в `CLAUDE.md` («Новая таблица с пользовательскими данными»).
Кратко:

- Любой **свободный текст пользователя** пишется/читается через
  `encryptRecord` / `decryptRecord` (`src/utils/crypto.ts`) с объявленной
  `EncryptSchema`. AES-256-GCM.
- **Не шифровать:** `id`, `userId`, `createdAt`, `updatedAt`, enum-поля
  (`needId`, `schemaId`, `modeId`) — по ним идут выборки и джойны.
- `ENCRYPTION_KEY` **нельзя ротировать без re-encryption-миграции** — иначе все
  зашифрованные данные станут мусором. План ротации — в `CLAUDE.md`.
- `decrypt()` поддерживает несколько ключей (fallback на старые) — это нужно
  для бесшовной ротации, не ломать эту логику.

## 6. Секреты и конфигурация

- Все секреты — через `ConfigService` или `process.env`. Хардкод запрещён.
- `.env` в `.gitignore` (проверено). Не коммитить, не вставлять в issue/чат.
- **Не логировать** секреты, токены, `initData` целиком, тело запроса целиком.
  В лог идёт причина и обрезанный контекст, не payload.
- Неиспользуемые секреты удалять из `.env` — каждый лишний секрет это лишняя
  поверхность утечки. (На 2026-05-20 в `.env` лежит мёртвый `THERAPIST_CODE`.)
- Критичные переменные и порядок их ротации при компрометации — в `CLAUDE.md`.

## 7. CSRF, cookie и CORS

- refresh-токен живёт в cookie: `httpOnly` + `secure` + `sameSite=strict` +
  `path=/api/auth`. Не ослаблять эти флаги.
- Любой state-changing эндпоинт, опирающийся на ambient-cookie (`refresh`,
  `logout`, `merge`, `telegram/widget`), требует `requireCsrf(req, ...)` —
  проверку заголовка `x-requested-with`, который кросс-доменная форма выставить
  не может.
- OAuth-флоу защищён cookie `oauth_state` (`sameSite=lax`, нужен для top-level
  редиректа от Google/VK) + сверкой `state`.
- Access-токен на вебе хранится **только в памяти** (React `useState`),
  **никогда** в `localStorage`/`sessionStorage`. localStorage уязвим к краже
  через XSS; httpOnly-cookie с refresh-токеном — нет.
- CORS: `origin` — явный список доменов, не `*` и не reflect заголовка `Origin`.
  В проде список задаётся `ALLOWED_ORIGINS`.

## 8. Анти-паттерны — так делать НЕЛЬЗЯ

- `delete({ where: { id } })` без `userId` в условии.
- `$queryRawUnsafe` / `$executeRawUnsafe` с конкатенацией строк. Только
  `Prisma.sql` с биндингом параметров (см. `merge.service.ts` как эталон —
  идентификаторы таблиц там идут через белый список `KNOWN_TABLES`).
- `JSON.parse(пользовательский ввод)` без `try/catch`.
- `Math.random()` для токенов, кодов, nonce — только `crypto.randomBytes`.
- `fetch(...)` к внешнему сервису без `AbortSignal.timeout(...)` — висящий
  сокет копится и исчерпывает ресурсы.
- `===` / `!==` для сравнения подписей и хешей (timing-атака).
- Поймать ошибку и вернуть `true`/успех (fail-open). Ошибка проверки доступа =
  отказ в доступе.
- Хранить любой токен в `localStorage`.
- `new Date(userInput)` без проверки формата.
- Эндпоинт без `@UseGuards`, «временно, потом добавлю».

## 9. Реагирование на инциденты

| Событие | Действия |
|---------|----------|
| Утёк `BOT_TOKEN` | revoke в BotFather → новый токен → env. Все `initData` инвалидируются, юзеры мини-аппа перелогинятся. |
| Утёк `JWT_SECRET` | ротация → все web-сессии и merge-токены мертвы, юзеры логинятся заново. |
| Утёк `ENCRYPTION_KEY` | ротация по плану из `CLAUDE.md`. Если утёк **вместе с бэкапом БД** — данные раскрыты, это критический инцидент. |
| Подозрение на захват аккаунта | `SecurityLogService` пишет `refresh_token_reuse` / `suspicious_initdata`; revoke сессий пользователя. |
| Всплеск `suspicious_initdata` | либо атака подбора подписи, либо ротация `BOT_TOKEN` не докатилась до всех инстансов. |
| Утёк дамп БД | данные зашифрованы AES-256-GCM; видны структура и enum-метки. Открыты только если `ENCRYPTION_KEY` утёк одновременно. |

Аудит-события (`merge_confirmed`, `role_changed`, `refresh_token_reuse`,
`csrf_blocked`, `suspicious_initdata`, …) шлются DM админу через
`SecurityLogService`. Их стоит читать.

## 11. Фронтенд (webapp и мини-апп)

React экранирует вывод по умолчанию, `dangerouslySetInnerHTML` в коде нет —
поверхность classic-XSS низкая. Но есть свои слабые места:

- **Клинический текст кэшируется в `localStorage` открытым текстом.** Письма
  себе, «безопасное место», черновики дневника, ответы YSQ, метки схем/режимов
  мирорятся в `localStorage` для мгновенного рендера. На бэкенде это всё
  шифруется AES-256-GCM — а в `localStorage` оно лежит **в открытом виде на
  диске устройства**. Частично закрыто: `logout()` чистит
  `localStorage`/`sessionStorage` (важно для общих устройств). Остаточный риск:
  пока юзер залогинен, любой XSS или доступ к профилю браузера = доступ ко всей
  клинической истории. На будущее — либо не кэшировать чувствительный текст
  совсем (лоадер до ответа сервера), либо шифровать кэш ключом, живущим только
  в памяти сессии.
- **Нет Content-Security-Policy.** `webapp/index.html` отдаётся без CSP. CSP —
  defense-in-depth: при XSS ограничивает, откуда грузятся скрипты и куда уходят
  данные. Включать осторожно (приложение на инлайн-стилях — обязателен
  `style-src 'unsafe-inline'`), проверив рендер в браузере. Лучше отдавать
  HTTP-заголовком, не мета-тегом. Стартовая политика:
  `default-src 'self'; script-src 'self' https://telegram.org;
  style-src 'self' 'unsafe-inline'; connect-src 'self' <VITE_API_URL>;
  img-src 'self' data:; base-uri 'self'; frame-ancestors 'self' https://*.telegram.org`.
- **Access-токен — только в памяти.** Хранится в React-state, не в
  `localStorage`. Refresh-токен — в httpOnly-cookie. После OAuth токен приходит
  во фрагменте URL и сразу вычищается (`AuthCallback.tsx`, `replaceState`).
  Эту схему не менять — `localStorage` для токенов запрещён.
- **`VITE_*`-переменные публичны.** Vite инлайнит их в бандл. Сейчас там только
  `VITE_API_URL` и `VITE_BOT_USERNAME` — не секреты. Секрет в `VITE_`-переменную
  класть нельзя — он окажется в открытом JS.
- **Клиентские флаги роли** (`therapist_mode` и т.п. в `localStorage`) — это
  только UX. Граница доступа — всегда сервер (`role` в БД + `assertRelation`).
  Не полагаться на клиентские флаги для безопасности.

## 13. Принятые риски

Осознанно не чиним — для текущего масштаба (российский хостинг, один инстанс,
небольшая аудитория) это accepted risk:

- **In-memory состояние на одном инстансе:** PKCE-верификаторы VK, стор
  throttler-а, `pendingPairCodes`. Горизонтальное масштабирование (>1 инстанс)
  их молча сломает — при переходе вынести в Redis.
- **Верификация Google ID-token через `tokeninfo`** (debug-эндпоинт Google,
  rate-limited). Для большого трафика — перейти на локальную проверку подписи
  по JWKS.
- **merge-токен и `link_token` в URL** (query/fragment) — попадают в логи
  и историю браузера. TTL 10/15 мин ограничивает экспозицию.
- **`VACUUM FULL` не выполняется** при удалении аккаунта — см. `CLAUDE.md`.
- **Бэкапы Amvera** — открытые вопросы к их саппорту перечислены в `CLAUDE.md`.

## 15. Security-трипваеры (тесты-храповики, 2026-07)

Инварианты безопасности закреплены исполняемыми тестами — регрессия
роняет CI, а не всплывает на проде. Все allowlist'ы **могут только
сокращаться**. Пробит по классам уязвимостей: реальных дыр в контуре не
найдено (код уже крепок), тесты фиксируют это и ловят будущие регрессии.

- **JWT-форджинг** (`src/auth/auth.service.security.spec.ts`): alg:none,
  смена алгоритма, type-confusion access/refresh/link/merge, подмена
  iss/aud, tamper — всё отклоняется; пин `algorithms:['HS256']`.
- **Обход авторизации** (`src/security/route-auth.invariants.spec.ts`):
  каждый контроллер под guard, либо admin-key на каждом методе, либо в
  явном публичном allowlist. Новый контроллер без защиты роняет тест.
- **Инъекции/RCE/pollution** (`src/security/injection.invariants.spec.ts`):
  бан eval/Function/child_process; `*RawUnsafe` только с литералом;
  `Prisma.raw` всегда через whitelist `ident()`; `__proto__` не загрязняет
  прототип.
- **Constant-time** (`src/security/timing-safe.invariants.spec.ts`): сайты
  сравнения секретов зовут `timingSafeEqual`, запрет `===` на секрете.
- **CSRF** (`src/security/csrf.invariants.spec.ts`): каждый читатель
  refresh-куки зовёт `requireCsrf`; куки не `sameSite:none`.
- **Hardening** (`src/security/hardening.invariants.spec.ts`): helmet+CSP,
  ValidationPipe whitelist, cap тела, exception-фильтры, HTTPS-редирект.
- **ReDoS** (`webapp/src/utils/crisisMarkers.redos.test.ts`): детект на
  патологическом вводе укладывается в 100мс.
- **initData-фаззинг** (`src/security/initdata-fuzz.spec.ts`): патологический
  user не даёт 500 и не загрязняет прототип.
- **XSS** (`webapp/src/security/xss.invariants.test.ts`):
  `dangerouslySetInnerHTML` только в allowlist и только с DOMPurify.
- **Capability-токены** (`src/security/capability-token.spec.ts`):
  `/by-token/:token` (booking/subscription) отдаёт строгий allowlist без
  PII; cancelToken = `randomUUID()`.
- **Утечка в логи** (`src/security/log-leak.invariants.spec.ts`): ни один
  logger-вызов не интерполирует токен/секрет/`decrypt()`/`req.body`.
- **SSRF** (`src/security/ssrf.invariants.spec.ts`): server-side `fetch`
  не строит URL из пользовательского ввода.
