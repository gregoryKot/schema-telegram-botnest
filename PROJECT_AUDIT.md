## 1. Безопасность — 8.5/10

Проверено вручную: `src/auth/*`, `src/utils/crypto.ts`, `src/api/telegram-auth.guard.ts`,
`src/api/throttler.guard.ts`, `src/therapy/therapy.controller.ts`, `src/auth/merge.service.ts`, `src/main.ts`.

### Что сделано хорошо (подтверждено чтением кода)

- **Крипто** (`src/utils/crypto.ts`): AES-256-GCM, случайный IV, `authTagLength` пиннован к 16, multi-key ротация с fallback, fail-loud на проде без ключа (процесс падает на старте — правильно).
- **Auth** (`auth.service.ts`): refresh — random 40 байт, хранение по хэшу, `timingSafeEqual`, ротация с token families (детект переиспользования), HS256 с пиннованными algorithm/issuer/audience — alg-confusion исключён.
- **Web user ID** — случайный BigInt в [1e15, 9e15): ниже 2^53, поэтому глобальный патч `BigInt→Number` (`main.ts:14`) не теряет точность. Граница выбрана сознательно.
- **Cookies**: refresh — httpOnly+secure+sameSite:strict; oauth_state/PKCE — httpOnly, 10 мин, path-scoped. OAuth со state+PKCE.
- **initData**: валидация подписи, expiresIn 1ч, `SKIP_AUTH` жёстко выключен в production независимо от env, алерт админу при невалидной подписи.
- **Authz терапевта**: каждый client-эндпоинт = роль THERAPIST + `assertRelation(status:'active')`. Ручное добавление реальных юзеров без согласия **осознанно отключено** (`therapy.controller.ts:86`) — правильное решение для терапевтического продукта. Роль — только через админ-одобрение.
- **Merge** (`merge.service.ts`): whitelist SQL-идентификаторов, WebSession уничтожается вместо переноса (иначе refresh старого аккаунта был бы валиден для нового), транзакция.
- **HTTP**: helmet+CSP, body cap 256KB, CORS-whitelist, canonical-host редиректы. Единственный `dangerouslySetInnerHTML` — через DOMPurify.

### Находки

**S-1 · Rate-limit обходится ротацией неверифицированных идентификаторов — Важно**
`UserThrottlerGuard.getTracker()` (`src/api/throttler.guard.ts`) бакетирует по `sub` из JWT
**без проверки подписи** и по `user.id` из initData **до** валидации (троттлинг идёт раньше auth-гарда).
Атакующий меняет `sub` в каждом запросе → свежий бакет на запрос, per-IP лимита нет →
brute-force-усиление по auth-эндпоинтам; плюс можно исчерпать бакет жертвы, зная её userId.
*Фикс*: для неверифицированных кредов бакетировать `uid|ip` либо падать на IP-бакет.

**S-2 · Реестры user-таблиц дублируются без совместной проверки — Важно (процесс)**
`USER_DATA_TABLES` (`bot.service.ts`, удаление аккаунта) и `USER_OWNED_TABLES`
(`merge.service.ts`, перенос при merge) — независимые списки одних и тех же моделей.
Чеклист CLAUDE.md упоминает только первый; ни один spec не сверяет их со `schema.prisma`.
Сценарий: новая модель попала в один список, но не в другой → при merge данные остаются
на старом userId (потеря для юзера + нарушение retention-логики).
*Фикс*: spec, который находит в schema.prisma все модели с `userId` и ассертит присутствие
в обоих списках; дописать шаг в чеклист CLAUDE.md.

**S-3 · `decrypt()` молча возвращает сырое значение при провале GCM-аутентификации — Незначительно**
`crypto.ts:69` — легитимно для legacy plaintext, но подделанный/побитый шифротекст молча
уйдёт юзеру как base64-мусор без лога о нарушении целостности. *Фикс*: если строка похожа
на наш формат (base64, ≥29 байт) и все ключи провалились — warning-метрика.

**S-4 · `link_token` в query-параметре — Незначительно**
`jwt.guard.ts:44` — токены в URL попадают в логи/историю. Смягчено TTL 60 с. Accepted risk
или короткоживущая cookie.

**S-5 · Нет CI — Важно**
144 юнит-теста существуют и проходят, но их никто не запускает автоматически; lint не
енфорсится. Регресс в auth/merge может уехать в прод молча. *Фикс*: GitHub Actions
`npm ci && tsc --noEmit -p tsconfig.build.json && jest` на каждый PR/push.

## 2. Качество кода: бэкенд — 6/10

### Соответствие собственным правилам (CLAUDE.md)

- **Лимит ~150 строк**: 22 из 97 файлов превышают — системно, не исключение.
  Топ: `therapy.service.ts` 889 (×5.9), `bot.service.ts` 713, `api.controller.ts` 712,
  `telegram.service.ts` 478, `bot.analytics.service.ts` 470.
- **Тесты**: 8 spec на 97 файлов. Без единого теста: `therapy.service.ts` (42+ метода,
  включая денормализованные инварианты), большая часть `bot.service.ts` (YSQ, pairs,
  practices, roles, `deleteAllUserData`), весь `booking/*` и `subscription.service.ts`
  (платёжная логика!), telegram-настройки/расписания.
- **Разделение Telegram/бизнес**: формально соблюдено (`bot.service` не импортирует telegraf).
- **`console.log`**: 0 в бизнес-коде. **try/catch в хендлерах**: все 34 обёрнуты.
- **`answerCbQuery` до БД — 1 нарушение**: `telegram.service.ts:279-303`
  (`treq:approve|reject`) — сначала `approve()`/`reject()` в БД, потом `answerCbQuery` →
  риск вечного спиннера у админа. Остальные ~23 хендлера корректны.

### Архитектура

- **God-объекты**: `bot.service.ts` — 55+ методов, ~13 доменов; `therapy.service.ts` —
  42 метода; `api.controller.ts` — 62 эндпоинта, 8 инжектированных сервисов.
- **Нет DTO-валидации (важно)**: `class-validator`/`ValidationPipe` отсутствуют. Все ~40
  `@Body()` — inline TS-интерфейсы (только compile-time). Ручная валидация разного
  качества по эндпоинтам, циклы валидации дублируются.
- **Дублирование**: `uid()`/`parseId()` скопированы в 3 контроллера **с расхождением
  поведения** (therapy-версия разрешает отрицательные id — нужно для virtual-клиентов,
  но нигде не задокументировано; diary-версия — `parseInt`, толерантный к `"5abc"`).
  `localDateString`/`userTimezone` задублированы в `bot.service.ts`/`bot.analytics.service.ts`
  при существующем едином `notification.time.ts`. Проверка `ADMIN_ID` — 7+ копий.

### Типизация

247 `any` в src (без auth). `therapy.service.ts` — 44, с систематическим кастом **всего
Prisma-делегата**: `(this.prisma.modeMap as any)`, `(this.prisma.therapistCustomMode as any)` —
типобезопасность для этих таблиц выключена полностью. `no-explicit-any` осознанно
отключён в eslint-конфиге; `no-floating-promises` — warn, не error.

### Prisma

- **N+1 (важно)**: `TherapyService.getClients()` (`therapy.service.ts:154-198`) — ~6 SQL
  на каждого клиента (3 аналитик-запроса + 3 повторных чтения таймзоны). 50 клиентов
  → ~300 запросов на один вызов. Чинится батчем `rating.findMany({ userId: { in } })`
  + одним чтением таймзон.
- **Read-modify-write без атомарности**: `addToMyList` (`bot.service.ts:500-508`) —
  гонка на `mySchemaIds`/`myModeIds` при конкурентных upsert (lost update);
  `saveConceptualization` (`therapy.service.ts:555-617`) — та же схема с history-массивом.
- Хорошие примеры транзакций есть (`saveChildhoodRatings`, `chargeDue`).

### Прочее

- 0 TODO/FIXME/`@ts-ignore` в коде — заглушек не забыто.
- `pendingPairCodes` — in-memory Map: теряется при рестарте, не шарится между инстансами
  (TTL 15 мин, некритично, но заметка на масштабирование).

## 4. Продукт глазами клиента и вовлечённость — 7.5/10

### Сильные стороны

- **Retention-инженерия выше среднего**: адаптивная частота напоминаний
  (`notification.cadence.service.ts` — 3 игнора подряд → тихое понижение, возврат →
  плавное повышение до потолка юзера), стрики + 13 достижений с шерингом,
  **weekly recap уже реализован** (тренды + практика под слабейшую потребность),
  win-back «без чувства вины» (`notification.templates.soft.ts`, value_recap на 14-й
  день — зеркало собственных данных, не generic-текст), парный трекинг.
- **Контент глубокий**: `needData.ts` — на каждую потребность question/examples/ranges/
  tips по уровням, ~30 уникальных формулировок на уровень, обе формы обращения.
- **YSQ (116 вопросов) продуман**: прогресс в localStorage + на сервере, «Продолжить (X из 116)».
- **Монетизация сдержанная и правильная по тону**: donate-nudge не раньше 3-го дня,
  не чаще раза в 30 дней, dismissible; кнопка записи к терапевту — только при
  устойчивом паттерне (10+ дней ниже порога), не рефлекторная продажа.
- Лендинг не обещает ничего, чего нет.

### Слабые места

1. **Кризисный guardrail отсутствует (главная находка аудита)**. Ни в боте, ни в
   мини-аппе нет ни детекции кризисных маркеров в дневниковых записях, ни автоответа
   с телефоном доверия. Номер 8-800-2000-122 существует в проекте — но только на
   лендинге терапевта (`LandingPage.tsx:868`), не в рантайме продукта. Пользователь
   может писать в зашифрованный дневник о суицидальных мыслях — продукт никак не
   отреагирует.
2. **Отложенный aha-момент**: продукт честно говорит «паттерн через 3–5 дней», но в
   первый визит нет ничего мгновенно-ощутимого (например, интерпретации «сегодняшнего
   профиля» одной фразой после первого заполнения).
3. **Онбординг бота — 2 текстовых экрана** (согласие + ты/вы) до полезного контента.
4. **Подписка готова на бэкенде, но скрыта** (`/subscribe` ждёт Robokassa recurring) —
   инфраструктура простаивает.
5. В паре виден только индекс напарника — нет лёгкого социального триггера
   (уведомления об активности), который усилил бы парный retention.

## Сводный план действий

1. **Кризисная карточка в продукте**: при сохранении дневниковой записи — детекция
   явных маркеров → неблокирующая карточка с 8-800-2000-122. Самый важный и дешёвый
   фикс всего аудита.
2. **CI**: GitHub Actions с `tsc + jest` (бэкенд) — полдня работы, закрывает S-5.
3. **`answerCbQuery` в `treq:approve|reject`** — поднять до БД-вызовов (5 минут).
4. **Троттлинг (S-1)**: бакет `uid|ip` для неверифицированных кредов.
5. **Точечные ты/вы-фиксы**: `BeliefCheck.tsx`, miniapp `HomeView.tsx:117`,
   `CheckInSheet.tsx` (оба), + `useHistorySheet` в `HelpSection.tsx`.
6. `eslint --fix` (снимает ~2000 prettier-ошибок автоматически).

### Важно (недели)

7. **Spec-сверка реестров таблиц (S-2)**: schema.prisma ↔ `USER_DATA_TABLES` ↔
   `USER_OWNED_TABLES` + шаг в чеклист CLAUDE.md.
8. **Тесты на `therapy.service.ts`** (в первую очередь `assertRelation`, streak-задачи,
   conceptualization history) и платёжный `subscription.service.ts`.
9. **N+1 в `getClients()`** — батчинг (заметно уже при десятках клиентов у терапевта).
10. **DTO + ValidationPipe** — вводить инкрементально, начиная с новых эндпоинтов.
11. **Мгновенный aha в первый визит** + объединить согласие и выбор ты/вы в один шаг.
12. Удалить мёртвые `webapp/.../diary/HomeView.tsx`/`DiaryListView.tsx` (345 строк),
    перенести рефакторинг `useAddClient`/`useClientDetail` в miniapp.

### Долгосрочно (месяцы)

13. **Shared-пакет** для `api.ts`/`types.ts`/`needData.ts`/`utils` двух фронтендов —
    устраняет корневую причину рассинхронизаций (уже 3+ подтверждённых случая).
14. Распил god-объектов (`bot.service`, `therapy.service`, `api.controller`) по доменам —
    по образцу уже вынесенных `diary.service.ts`/`profile.service.ts`.
15. Включить подписку (бэкенд готов), эскалация тревожной записи терапевту одним тапом,
    уведомления об активности напарника в паре.
16. Гранулярные ErrorBoundary в miniapp; ретраи/очередь оффлайн-запросов в api-клиентах.
